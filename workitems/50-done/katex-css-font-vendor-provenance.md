# Standardize KaTeX CSS and font vendoring provenance

## Summary

The vendored dependency workflow already standardized KaTeX JS, but the CSS and
font assets still lacked explicit provenance.

That gap is now closed.

The current KaTeX CSS/font inputs are sourced from the pinned installed package:

- `node_modules/katex/dist/katex.min.css`
- `node_modules/katex/dist/fonts/*.woff2`

They are now regenerated by a reproducible dedicated asset-vendor script that:

- rewrites upstream font URLs for PiClaw's static asset layout
- keeps only the shipped `woff2` font files
- emits committed provenance metadata covering the CSS plus all exported fonts

## Acceptance Criteria

- [x] Document where the current KaTeX CSS and font assets originate from
- [x] Decide whether they should be copied from `node_modules/katex/dist/` or rebuilt/generated another way
- [x] Add reproducible metadata / provenance for the CSS and font assets
- [x] Ensure the web build continues to produce working KaTeX rendering after the provenance cleanup
- [x] Record whether the CSS/fonts belong inside the shared vendored-dependency workflow or a parallel static-asset workflow

## Implementation Decision

### Chosen workflow

Use a **parallel static-asset workflow**, not the existing JS-centric manifest
bundle workflow.

### Why

KaTeX CSS/fonts are not a straight JS bundle or a raw file-copy case:

- the upstream CSS references `fonts/...` paths that must be rewritten for
  PiClaw's checked-in static layout
- PiClaw intentionally ships the `woff2` subset rather than the full
  `woff` + `ttf` fallback matrix from upstream dist
- the generated CSS must remain at `web/src/styles/katex.bundle.css` so the web
  build continues to import it through `web/src/styles/app.css`

That makes this a better fit for a dedicated reproducible asset script than for
forcing it into the existing JS manifest path without additional transform
abstractions.
